二、当前设计审阅
2.1 核心抽象总结
EventSignal<Args...> / Connection / ScopedConnection

Reactive.ixx §1，约第 52–231 行

    解决的问题：多播事件（UI 点击、信号触发）的线程安全订阅与生命周期管理。
    设计亮点：
        emit() 先在锁内快照 slot 列表，再不持锁调用回调，避免死锁（emit() 可嵌套调用 connect/disconnect）。
        connect_once 一次性订阅语义清晰。
        ScopedConnection RAII 封装，在 DemoPage 中作为 prop_conn_、list_watch_conn_ 成员存活，生命周期对齐。
    隐患：on_click() 等 Widget API 内部使用 std::vector<std::function<void()>> 直接存储回调，而非返回 Connection，这是有意的异化——按钮注册回调应该"永久有效"。文档（some_issue.md §5）已记录这个决策，但它与 EventSignal 的"需显式保存 Connection"语义不一致，是认知落差的来源之一。

State<T> / ReadState<T>

Reactive.ixx §2，约第 261–354 行

    解决的问题：可观察值。写值通知观察者（Effect 重运行 + persistent on_change 回调）。
    设计亮点：
        非拷贝、非移动，防止裸指针悬垂（ReadState<T> 持有 const State<T>*）。
        notify() 先移出所有 ObserverEntry 再触发，允许 Effect 在回调内重新注册（防止迭代器失效）。
        双路通知：Effect 追踪路径（observers_）+ 持久订阅路径（change_signal_）。
    隐患：
        State<T> 要求 T 实现 operator==。若 T 无 ==，set() 不编译。未用 concept 约束，错误信息会较难读。
        observers_ 是 mutable std::vector<ObserverEntry>，无锁保护。EventSignal 有锁，但 Effect 追踪通路无锁——目前仅假设单线程使用，文档中亦有说明，但两个子系统的线程安全基线不对称，后续多线程扩展时容易出现不一致预期。

Computed<F>

Reactive.ixx 约第 357–400 行

    解决的问题：惰性派生值，自动追踪依赖，访问时按需重算。
    设计亮点：lazy re-evaluate（stale_ 标记），支持 CTAD (Computed sum{[&]{...}}）。
    隐患（潜在 UAF）：Computed 是可移动的（default move）。recompute() 中的本地 lambda [this]{ stale_ = true; } 的地址在每次 recompute() 时重新创建，拷贝给 State 的 ObserverEntry。若 Computed 在首次求值后被移动（例如 std::vector<Computed<...>> 扩容），已存储在某个 State 的 ObserverEntry.invalidate 仍持有旧地址的 lambda，其中的 this 指向已析构对象——UAF / 行为未定义。当前代码中 Computed 均为具名成员变量，未移动，所以未触发。但 API 上允许移动，是个隐藏炸弹。

Effect / EffectScope

Reactive.ixx 约第 401–485 行

    解决的问题：副作用与 State 依赖的声明式绑定，自动重执行，RAII 生命周期。
    设计亮点：run() 每次重运行时重新创建 self_invalidator_，捕获当前 this，确保自身地址的 lambda 始终有效（只要不移动）。active_ 标记析构时置 false，防止析构后回调。
    隐患（EffectScope 扩容 Bug）：

        EffectScope 内部是 std::vector<Effect>。

        每次 add(fn) 调用 effects_.emplace_back(fn)，在构造新 Effect（立即执行 run()）时若 vector 扩容，已存在的 Effect 对象会被移动构造。

        Effect 的移动构造器（Reactive.ixx 约第 415–423 行）：self_invalidator_ 被 move 过来（仍捕获旧的 this），other.active_ = false。

        此后，State 触发旧 invalidator → old_effect.run() → active_= false → 提前返回。新地址处的 Effect 永久失去响应。

        实际影响：当前所有组件每个 scope_ 最多注册 1 个 Effect（Label 1个，CounterPage 1个），从未触发扩容。但这是一个等待爆炸的定时炸弹。

        代码路径：EffectScope::add() → effects_.emplace_back() → vector reallocate → Effect::Effect(Effect&&) → other.active_ = false。

Prop<T>

Reactive.ixx §3，约第 487–554 行

    解决的问题：组件属性统一类型——使调用者不必区分"静态值"和"响应式绑定"。
    设计亮点：std::variant<T, State<T>*> 简洁，get() 透明转发追踪上下文，on_change() 对静态 prop 返回 no-op Connection（连通性判断清晰）。
    隐患/设计不完整：
        LabelProps（Label.ixx 第 15 行）仍然是 const State<std::string>* text_signal = nullptr，没有用 Prop<std::string>。Prop 没有真正落地到组件 API。
        Button 的 text_ 完全不响应式，没有 Prop 封装。
        Prop 的 on_change() 是 EventSignal-based 持久订阅，DemoPage 就是这么用的；但 Prop 的 get() 在 Effect 上下文中调用时是 Effect-tracked 的。两条路径并存，文档中没有明确区分"什么时候用哪条"，容易造成混乱。

StateList<T>

Reactive.ixx §4，约第 557–647 行

    解决的问题：可观察列表，细粒度 diff 事件（Inserted/Removed/Updated/Moved/Reset）。
    设计亮点：ListChangeKind 枚举 + ListChange<T> 描述符设计完整，既支持细粒度（on_change）也支持粗粒度（watch）订阅，适合列表渲染优化。
    隐患：
        StateList 完全不参与 Effect 自动追踪系统。无 detail::current_tracking_context 调用。这意味着在 Effect 中读取 list.size() 或 list[i] 不会建立依赖，不会自动重触发。使用者必须手动 on_change/watch 订阅。
        这造成了双系统割裂：State 用 Effect 自动追踪，StateList 必须手动订阅。开发者面对两套思维模型。

2.2 职责划分评估
概念	职责	重叠/混淆风险
State<T>	单值响应式容器	与 Prop 有语义重叠（Prop 不新增语义，只做类型统一）
ReadState<T>	父→子只读传递	与 const State<T>* 无本质区别，薄包装
Prop<T>	静态/响应式属性统一 API	目前未被实际组件 Props 采用，是"存在但未推广"的抽象
StateList<T>	列表状态	与 Effect 系统完全割裂，形成双订阅模型
Computed<F>	派生状态	与 Effect 相比：Computed 是懒求值只读派生，Effect 是主动副作用
Effect / EffectScope	副作用 + 生命周期	职责清晰，但与 on_change 存在功能重叠

总体评判：核心响应式三元组（State / Effect / EffectScope）职责清晰、边界明确。Prop 是个合理的方向但落地不足。StateList 与 Effect 的分离是一个有意识的架构选择，但目前缺乏足够文档解释，给使用者带来认知负担。
2.3 与 UI 组件系统的契合度

当前路径（State → Effect → mark_dirty() → 重渲染）：
Code

State<int> count_ → Effect [在 build() 中注册] → 读 count_() → count_ 触发 notify()
    → self_invalidator_() → Effect::run() → label_ptr->text(format("{}",count_()))
    → Label::text() → text_.set(...) → Label 内部 Effect 触发 → mark_dirty() → 冒泡
    → render_widget() 剪枝

这条路径功能完整但链路较长：用户 State 变化 → 外部 Effect → 手动调 widget 方法 → 内部 State 变化 → 内部 Effect → dirty 标记。两层 Effect + 两层 State。

更直接但不安全的路径（Label::Create(LabelProps) 方式，Label.ixx 第 26–36 行）：
C++

label->scope_.add([lbl = label.get(), sig]{
    lbl->text_.set((*sig)());  // 但 sig 可能是 State* 而非 Prop*
});

这里用裸指针捕获，且直接在 scope_.add 中操作。DemoPage 使用的是更安全的 on_change + ScopedConnection 路径。

评估：与 dirty-propagation 机制的对接是工作的，但需要两次 Effect 跳转。理想的架构应该将 bind_text(state, transform) 等高层 API 内化为框架能力，而不是依赖使用者手写 Effect + 裸指针捕获。
2.4 API 一致性评估

✅ 好的部分：

    State<T>: operator()() 读，.set() 写，一致性强。
    EffectScope::add() 与 CompositeComponent::effect() 风格统一。
    Connection / ScopedConnection RAII 模式清晰。
    StateList 的 API 命名（push_back/remove_at/update_at/move_item）符合直觉。
    Computed CTAD deduction guide 设计良好。

⚠️ 值得警惕的部分：

    Prop<T> 与组件参数系统（LabelProps）未对齐，形成两套传参风格（Prop<T> vs 裸指针）。
    Button::text() 返回 Button&（fluent），但 Label::text() 也返回 Label&——风格相似但 Label::text() 调用的是 text_.set()（响应式），Button::text() 直接赋值给 std::string text_（非响应式）。同名方法语义不等价。
    ScopedConnection 在 DemoPage 中被显式作为成员变量保存，但 Button::on_click 不需要保存返回值。来自 some_issue.md §5 的这个不一致已被记录，但还未通过框架层统一解决。

四、下一步开发建议
4.1 近期必须做（稳定基础）

P0：修复 EffectScope 扩容 Bug

EffectScope 的 effects_ 改为 std::deque<Effect> 或 std::list<Effect>（不发生移动）；或在 add() 中先 effects_.reserve() 一个足够大的值；或根本方案是将 Effect 设计为不可移动并用 unique_ptr<Effect> 管理：
C++

// 方案A：用 deque（最小改动，但 deque 不能保证地址稳定）
// 实际上 deque 扩容时端部追加无需移动中间元素，但 push_front 除外

// 方案B（推荐）：heap-allocate Effect
class EffectScope {
    std::vector<std::unique_ptr<Effect>> effects_;
public:
    template<typename F>
    auto add(F&& fn) -> void {
        effects_.push_back(std::make_unique<Effect>(std::forward<F>(fn)));
    }
};

这样 vector 扩容只移动 unique_ptr，Effect 对象地址不变。

P0：修复/限制 Computed 移动语义

Computed 要么改为不可移动（= delete 移动操作），要么让 invalidator 不依赖 this 指针地址（例如通过 shared_ptr<bool> stale flag 共享语义）。最简方案：
C++

Computed(Computed&&) = delete;
auto operator=(Computed&&) = delete;

使其与 State 同等非移动，消除 UAF 风险。

P1：同步里程碑文档

将 docs/design.md 中 M1 状态从 "🚧 进行中" 更新为 "✅ 完成"（M1 全部 checklist 已 [x]）。将 src/reactive/ 下未编译的拆分文件标记为 [legacy, not compiled] 或清理。
4.2 中期应该做（API 收敛）

决策点：Prop<T> 推广还是收缩？

    这里有两条路线，需要架构决策：

    路线 A（推广 Prop）：将 LabelProps.text_signal 改为 Prop<std::string>，Button 的 text() 也支持 Prop<std::string>，统一所有组件属性为 Prop 体系。优点：API 一致性高，消除裸指针传参。缺点：Prop 本身是浅封装，增加一层但语义增益有限。
    路线 B（简化为直接 State&）：废弃 Prop，改为组件直接接受 State<T>& 或 const State<T>&，静态值通过默认构造的 State<T> 传入。优点：减少抽象层数，语义直接。缺点：调用方必须总是为静态值也创建 State 对象。

建议：推进路线 A，但同时补充高阶绑定 API，减少手写 Effect 的需要：
C++

// 高层绑定 API（在 CompositeComponent 或 Label 内实现）
label->bind_text(count_, [](int v){ return std::format("{}", v); });
// 内部 = scope_.add([label, &count_, transform]{ label->text(transform(count_())); })

StateList 与 Effect 系统的定位决策

    两条路线：

    路线 A（保持分离）：StateList 只用 EventSignal，Effect 只用 State。文档中明确说明"列表变更用 on_change，值变更用 Effect"。优点：列表 diff 语义精确，避免整体重渲染。
    路线 B（StateList 接入 Effect 追踪）：在 StateList 的 size()、operator[] 等读方法中调用 detail::current_tracking_context，允许在 Effect 中读取列表触发自动订阅。优点：使用统一的响应式模型。缺点：只能粗粒度通知（整个 Effect 重跑），失去 diff 优势。

建议：路线 A，但需要在文档和组件模式中更清晰地区分两个模型，避免开发者困惑。可以为 StateList 添加一个 as_state(transform) 桥接方法（将列表的某个派生值包装为 Computed<F>）。

Button 的响应式化

当前 Button 的 text_ 是普通 std::string，不响应式。至少应支持 bind_text(State<std::string>&)，以便动态按钮文本（如"保存中..."/"已保存"）的实现。

ScopedConnection vs Effect 的使用规范化

提供明确的框架级准则（在文档或 API 注释中）：

    描述 UI 属性与 State 绑定：用 effect() + 内部 State 方式，或用高阶 bind_* API
    订阅外部事件（Signal）：用 ScopedConnection
    订阅 StateList：用 on_change + ScopedConnection

4.3 远期可考虑

Effect 批处理 / transaction
C++

Nandina::batch([]{
    count_.set(1);
    name_.set("x");
}); // 合并为一次 Effect 重运行

细粒度组件局部更新

当前所有 dirty 均冒泡到根节点，渲染剪枝只在没有 dirty 子树时跳过。长远看，如果需要支持大型组件树，应考虑局部重建（类似 React 的 fiber 思路）而非全树脏检查。

调试能力

    Effect 应支持命名（scope_.add("count_label", [...])）以便调试输出
    State 应支持 on_debug([](const T& old, const T& new){ ... }) 或 debug_name()

测试基础设施

some_issue.md §6 已指出：State/Effect/Computed 核心需要单元测试。重复读取、嵌套 Effect、生命周期销毁、EffectScope 多 Effect 均需覆盖。这也是发现扩容 Bug 的正确途径。

API 冻结时机

建议在 M2（布局系统）完成后，对以下 API 进行一次冻结讨论：

    State<T>::set() / get() / operator()() / on_change() → 稳定

    Prop<T> → 推广到组件系统后冻结
    StateList<T> 的读/写 API → 稳定，但订阅路径待统一
